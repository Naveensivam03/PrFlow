# Database Architecture Analysis - PRFlow

## Overview
PRFlow utilizes a PostgreSQL database managed via Flyway migrations. The schema is designed to separate **Workflow Memory**, **Organizational Memory**, and **Intelligence Enrichment**.

## 1. Entity Relationship Summary

### Organizational Memory (Core Graph)
These tables represent the static/semi-static structure of the organizations using PRFlow.

*   **`organizations`**: The root tenant.
    *   `github_org_id`, `github_installation_id` (Unique identifiers).
    *   Ownership: Owns all developers and repositories.
*   **`repositories`**: Monitored codebases.
    *   `github_repo_id` (Unique identifier).
    *   `expertise_mode`: Defines how expertise is calculated for this repo.
*   **`developers`**: Individual contributors.
    *   `github_user_id` (Unique identifier).
    *   Attributes: `review_capacity`, `reliability_score`.
    *   *Note: Currently only persisted when a developer authors a PR.*

### Workflow Memory (Event-Driven State)
These tables store the raw and processed state of pull requests.

*   **`pull_requests`**: Central entity for PRFlow orchestration.
    *   Columns: `status` (OPEN, CLOSED, MERGED), `complexity_score`, `complexity_level`.
    *   Foreign Keys: `repository_id`, `author_id`.
*   **`pull_request_files`**: Granular change data.
    *   Stores `file_path`, `scope_type`, `scope_identifier`.
    *   Aggregates `lines_added`, `lines_deleted`.
*   **`webhook_logs`**: Audit trail for incoming GitHub payloads (idempotency support).

### Intelligence Enrichment (Derived Knowledge)
Knowledge generated by orchestration engines.

*   **`developer_file_expertise`**: Maps developers to specific files/scopes.
    *   `expertise_score`: A decayed score based on historical contributions.
    *   Updated by the **Expertise Engine**.

## 2. Orchestration Responsibilities

| Table | Ownership | Responsibility |
| :--- | :--- | :--- |
| `organizations` | Ingestion Service | Tenant identification and isolation. |
| `repositories` | Ingestion Service | Resource scoping for analysis. |
| `developers` | Ingestion Service | Identity mapping (currently limited to PR authors). |
| `pull_requests` | Ingestion Service | Workflow state tracking (Persistence Layer). |
| `pull_request_files` | Ingestion Service | Signal extraction for engines. |
| `complexity_score` | Complexity Engine | Derived intelligence stored on the PR entity. |
| `developer_file_expertise` | Expertise Engine | Long-term organizational memory of who knows what. |

## 3. Critical Observations
*   **Missing Contributor Visibility**: There is no `repository_members` or `repository_developers` table to explicitly model which developers belong to which repository. Visibility is currently implicit through PR authorship.
*   **Knowledge Graph Gaps**: The "Knowledge Graph" is currently a tree: `Org -> Repos` and `Org -> Developers`. The connections between Developers and Repositories are only built as PRs are opened.
*   **Future Dependencies**: The **Assignment Engine** will require a `repository_developers` relationship to identify the pool of potential reviewers who are *not* the author.
